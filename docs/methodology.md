# 复杂技术文章的可视化方法论

## 为什么可视化

两个认知科学基础：

**Dual Coding Theory（Paivio, 1971）**
文字和图像激活不同认知通道，双通道同时激活 = 理解更深、记忆更牢。

**Cognitive Load Theory（Sweller, 1988）**
工作记忆容量有限。复杂文章负荷高的原因往往不是内容难，而是「结构不可见」——读者要一边读一边在脑子里建图。可视化的本质是**把建图的工作外包给界面**。

## 不是所有文章都适合

可视化的文章价值来自**结构和信息密度**，不是语言本身。

| 适合 | 不适合 |
|---|---|
| 技术教程、调研笔记 | 散文、小说 |
| 有层级/步骤/数据的内容 | 论证密度极高的哲学/法律文本 |
| 读者需要反复查阅的参考资料 | 情感型内容 |
| 结构复杂但语言是工具的文章 | 语言本身是价值载体的文章 |

## 结构类型 × 形式匹配

一篇文章内部可以有多种结构，应分块匹配，而不是整体选一种形式。

### 层级型 → 思维导图

信号：有章节嵌套、从属关系、"包括"/"分为"等
首选格式：**Markdown 缩进列表**——AI 生成稳定、人类可读、可直接被 Markmap 渲染为交互图
备选格式：otml（幕布专用导入）、Mermaid mindmap（掘金/GitHub 原生渲染）
适合受众：想系统理解全局的读者

### 数值型 → 图表 / 可交互计算器

信号：有数字对比、成本/性能数据、优化前后的量化结果
工具：Chart.js、D3.js、vanilla HTML+JS
适合受众：想看"到底省多少钱"的实用主义读者

### 时序型 → 流程图 / Scrollytelling

信号：有步骤编号、实施顺序、"先做A再做B"的决策链
工具：Mermaid、reveal.js + scroll-driven animations
适合受众：想被带着走、按顺序落地的读者

### 对比型 → 对比表格 / 矩阵

信号：有"A vs B"、取舍分析、多方案评估
工具：HTML table、矩阵图
适合受众：正在做决策的读者

### 关系型 → 知识图谱

信号：多个概念相互引用、系统组件依赖、非线性结构
工具：Obsidian Canvas、D3 force graph
适合受众：想理解概念间关系、而非线性阅读的读者

## 受众类型

同一个工具服务两种用户，需求不同：

| 类型 | 场景 | 动机 | 对工具的期待 |
|---|---|---|---|
| **读者型** | 看到别人的好文章，想帮自己或朋友理解 | 辅助理解、整理笔记、转发分享 | 快速、零门槛、产出物能直接发出去 |
| **创作者型** | 写自己的文章，想让读者更容易消化 | 提升传播效果、建立个人品牌 | 产出质量高、适配多平台、可迭代 |

两种受众都被服务，但对"分享路径"的需求不同：
- 读者型：发给朋友即可 → sharehtml.zhenjia.dev 链接（7天有效）
- 创作者型：发布到平台 → 嵌入博客或平台草稿

## 平台 × 形式匹配

不同平台的读者期待不同，同一篇可视化内容需要不同入口：

| 平台 | 读者期待 | 最适合的形式 |
|---|---|---|
| 微信公众号 | 有共鸣的叙事 | 配图 + 思维导图截图 |
| 掘金 | 干货 + 工具链 | 交互 HTML / 图表 |
| dev.to | 英文 walkthrough | Repo + 工具演示 |
| LinkedIn | 职业叙事 | 方法论总结卡片 |
| Twitter/X | 观点碎片 | 九图 / Thread |
| 个人网站 | 完整展示 | Scrollytelling |

## 可视化会失去什么

诚实地说：每种可视化都有代价。

- 思维导图：失去论证的推进感，只剩骨架
- 图表：失去上下文和注意事项
- 九图卡片：失去细节，容易被断章取义（洗稿的人就是这么干的）
- Scrollytelling：失去读者自由跳读的能力

**原则：可视化是原文的补充，不是替代。** 始终保留原文链接。
