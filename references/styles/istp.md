# ISTP Style — 机制拆解图

**认知偏好**：拆解机制 · 实用主义 · 极简直达 · 手到解决

**信息组织方式**：机制拆解式——直接切入「这个东西是怎么运转的」，用最少的层级把内部机制拆开，不解释背景，只看原理。

---

## 核心美学

极简的机制拆解图。像一位工程师在白板上快速画的「这东西是怎么运作的」——剥掉所有装饰，只留下核心机制和关键连接。
不解释为什么重要，不给出结论，只展示「这是结构」。读者自己判断。

---

## 设计 Token

```
--bg:        #F9F9F8   /* 近白 */
--surface:   #F0F0EE   /* 极浅灰 */
--text:      #1A1A18   /* 近黑 */
--text2:     #888880   /* 次要文字 */
--border:    rgba(0,0,0,0.14)
--mech:      #1A1A18   /* 机制连线和节点，同文字色 */
--key:       #D44000   /* 深橙红，唯一强调——关键机制节点 ONLY */
--key-bg:    #FFF0EA
```

颜色原则：**近乎无色 + 一点深橙红**。深橙红只标注「关键机制」节点，其余全灰黑。

---

## 布局原则

**机制拆解 + 组件关系**，像电路图或机械草图：

```
[输入条件]
    ↓
[组件 A] ──连接──→ [组件 B]
    ↓                  ↓
[组件 C]           [关键节点] ← 深橙红，最重要
    ↓
[输出结果]
```

- 节点：方形或矩形，无填充或极浅灰，近黑细边框
- 连线：细直线，有箭头方向
- 关键节点：深橙红边框，标注「关键」原因
- 无多余装饰，无背景色块
- 标注简洁：动词短语，不写完整句子

---

## 排版

- 节点文字：短语，3-5 字，等宽字体感
- 连线标签：极短（「→ 触发」「× 失效」「= 结果」）
- 关键标注：深橙红小字，解释「这里是关键」
- 无结论段落——拆解本身就是答案

---

## 特有视觉元素

- **关键节点**：深橙红边框 + 小字注释，是整张图最显眼的地方
- **组件标注**：每个节点下方有一行「作用：……」的小字说明
- **失效路径**：用虚线 + `×` 标注「当 X 不成立时，这条路径失效」
- **极简留白**：连线之间有足够呼吸空间，不堆叠
- **无结论**：图画完就结束，不加结语

---

## 收口方式

**无收口**——或者最多一行技术注释：

- 「以上为当前观测到的机制，未验证的路径已标注」
- 图直接结束，不加「因此……」或「建议……」

ISTP 读者不需要结论，他们**已经在看图的过程中自己得出结论了**。

---

## 不适合

- 情感色彩（那是 INFP/ESFP）
- 叙事弧线（那是 ENFP/ENFJ）
- 强结论（那是 ENTJ）
- 过多文字（一旦需要很多文字解释，说明图本身没有画清楚）
- 装饰性元素

### 签名视觉元素（v2）— 必须实现的布局语法

**① 机制节点（Mech Node）**
- 白底 + `1px var(--border)` 边框，`border-radius: 3px`，等宽字体感（12px bold mono）
- 节点下方可附 `node-sub`：10px `--text2`，说明作用
- 关键节点：`1.5px var(--key)` 橙红边框 + `--key-bg` 背景 + `--key` 文字色

**② 流程箭头（Arrow）**
- `flex: 1` 水平线，`1px --border`，右端三角箭头（`:after` 实现）
- 可选 `arrow-label`：9px mono，绝对定位在线上方，说明触发条件

**③ 关键注释行（Key Note）**
- 左侧 `2px var(--key)` 橙红边框 + `--key-bg` 背景块
- 「关键」小标签（10px bold mono `--key`）+ 说明文字（11px 正文）
- 解释关键机制的核心矛盾或约束

**④ 失效路径（Fail Path）**
- 顶部 `0.5px dashed --border` 分隔，`×` 符号（`--key` 色）+ mono 小字说明
- 表达「当 X 不成立时，这条路径失效」
- **无结论句**，图画完即止
