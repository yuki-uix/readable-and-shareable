# Digest — 怎么优化 Coding Agent 的成本：从 Prompt Cache 到 Context Engineering

**来源**：ThoughtWorks 内部博客大赛（Zhenjia Zhou, 2026-05-24）  
**日期**：2026-06-03

---

## 核心主张

> 作者真正在说的不是「省钱技巧清单」，而是一个认知重排：**AI coding 的成本不在于你用了多少功能，而在于你有多少重复计算**。每次 API 调用都要重新 Prefill 整个上下文——这是架构决定的，不是意志力能解决的。因此「用得少」不如「用得稳」：让前缀固定、让噪音出去，比少发一条消息便宜得多。

---

## 隐藏假设

1. **70% cache 命中率是可达的** — 全文成本计算的基础假设，但作者自己也承认是估算。实际命中率高度依赖前缀稳定性，一个 MCP 工具注册顺序随机就能让命中率归零。
2. **语义压缩不破坏质量** — DCP 等工具让 LLM 判断「这段对话是否完成」来决定是否压缩。边界情况下误判会悄悄丢失关键决策上下文，且事后无法感知。
3. **拆任务 = 省钱** — 文章花了大量篇幅介绍 Isolate 策略，但 UIUC 研究数据显示 multi-agent 消耗比 single-agent 多 4-220 倍。拆法不对时，比不拆更贵。

---

## So What

这篇文章最重要的认知更新不是「要用 Prompt Cache」，而是理解**为什么 input 占了 93-99% 的成本**：因为每轮 API 调用都要重发整个上下文（system prompt + CLAUDE.md + 对话历史 + 读过的文件），而 output 每次只生成几百 token。优化方向因此完全不对称——output 单价是 input 的 5 倍，但优化 input 的收益远大于优化 output。

落地顺序比工具选择更重要。作者的 6 步路径体现了一个反直觉的洞察：**最高 ROI 的动作往往是「配置类」而非「工具类」**——固定前缀顺序、关掉 attribution header、清理不用的 MCP，这些一次性配置能持续受益，而装一个新工具可能反而增加前缀 token。

---

## 盲点 / 反例

- **Context drift 是全文最大的未解问题**。代码在变、CLAUDE.md 在滞后，最终 Agent 在错误前提下工作。作者明确把这列为 open problem，但没有可操作的解法。
- **所有数字是粗算**。$11.30 → $4-5 的路径基于合理假设（70% cache 命中率等），不是实测结果。作者自己说了，但读者容易把估算当基准。
- **工具生态太新**。多数工具 3-6 个月历史，在大型生产代码库里的稳定性未知。RTK 在 monorepo 中只省了 1.6%，和宣传的 80% 差距巨大。

---

## 更大的连接

这篇文章和传统数据库优化路径惊人相似——索引（Select）、压缩（Compress）、分库分表（Isolate）、物化视图（Write）。LLM 时代的「DBA」，就是能做好 Context Engineering 的工程师。

更大的图景：这不是个人技巧问题，而是整个 AI 工具链的成本结构问题。GitHub Copilot 转 usage-based billing 是起点，不是终点。未来每个开发团队都需要有人专职管理「AI 上下文卫生」，就像现在有人专职管 CI/CD 一样。
