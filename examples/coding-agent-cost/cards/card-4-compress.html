Compress：减少已有上下文的 token 数
— 预防膨胀 vs 主动压缩，二者不可兼得
核心框架
压缩和缓存是对立的——预防比压缩更重要
压缩需要改历史才能缩短，缓存要求历史不变才能命中。两者不能同时最优。Context 占到 70–80% 时 Agent 就开始走神，不要等到 95% 才压缩——策略上预防膨胀远优于事后处理。
核心约束关系
预防类工具
不动已有历史
Cache 完全兼容，但只管新增内容
压缩类工具
修改 session 历史
Cache 归零，每次都需重新 Prefill
工具选型：预防类 vs 压缩类
预防类（不破坏 Cache）
RTK 80% 压缩
CLI 输出语义改写。Git/测试压缩率最高（92%/90%）。装完即用，一行命令。只管 Bash 输出。
lean-ctx 60–95%
Shell + Read + MCP 三通道压缩。10 种文件读取模式（map/signatures/diff…）。cached re-read 返回 ~13 tokens。
Headroom 87%
唯一主动考虑 cache 对齐的工具。CacheAligner 确保压缩后前缀与上次请求对齐，提高命中率。
压缩类（会破坏 Cache）
DCP ~85% cache hit
拦截 API payload，在发送前选择性替换完成的工具调用。原始 session 完整保留可回溯。
Cozempic 23–45%
直接操作 session JSONL 物理删除垃圾数据。18 种清理策略（Gentle/Standard/Aggressive）。
Magic Context 异步
后台 historian 异步压缩，主 Agent 不受打断。Facts 持久化跨 session，cache-aware 触发。
∴ 演绎结论
CLI 多装 RTK，大量文件读取用 lean-ctx，不想操心就装 Magic Context
Headroom 适合对 cache 命中率有极致要求的场景；DCP 适合想在 session 中途精细控制但不破坏本地日志的团队
